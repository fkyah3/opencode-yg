# 工具调用输出语言漂移分析

> 2026-04-27 记录
> 从实验中发现的关键限制

---

## 核心问题

"请用中文语言思维方式来完成所有任务"在长对话中保持率约 90%，但无法达到 100%。根因是：

**工具调用是底层代码执行，不经过 AI 模型调用，返回的是原始输出（英文）。**

每一次 bash、grep、read、edit、LSP 调用都在往上下文里灌入大量英文内容（文件路径、类型定义、代码片段、错误消息）。这些英文内容占据了约 50% 的上下文空间，持续牵引 AI 的语言倾向。

## 工具调用链路

```
用户消息 → AI 模型处理
  → AI 选择工具（如 bash/grep/read）
  → tool.execute.before (Hook，不改变调用)
  → **工具底层执行**（纯代码，无 AI 参与）
    → 执行 shell 命令 → 返回原始输出（英文）
    → 读取文件内容 → 返回原始内容（英文）
    → LSP 诊断 → 返回英文消息
  → tool.execute.after (Hook)
  → 输出返回给 AI 上下文
  → **AI 看到英文输出 → 语言权重偏移**
```

工具输出的语言**不受系统提示词控制**，因为工具本身不是 AI。它们是操作系统底层调用、Node.js API、第三方工具的直接返回。

## 关键层面对比

| 层级 | 语言 | 可控性 | 当前状态 |
|:----|:-----|:-------|:--------|
| 系统提示词 | ✅ 中文 | 高 | 愚公 212 行哲学框架 |
| 工具描述 | ✅ 中文 | 中 | 17 个工具已汉化 |
| 工具输出内容 | ❌ 英文 | **低** | 代码/路径/命令结果不可控 |
| 工具输出错误 | ❌ 英文 | 中 | T0 已汉化 6 处 |
| LSP 诊断 | ❌ 英文 | 低 | 依赖 LSP 协议 |
| 构建/测试输出 | ❌ 英文 | 低 | 外部工具输出 |

## 两种猜想

### 猜想 A：英文输出稀释中文密度（已验证 ✓）

工具输出的英文内容占据大量上下文空间，稀释了中文的"环境对齐"效果。这是已知有效的解释。

### 猜想 B：工具调用的 API 响应不附带提示词（需验证）

每个工具调用是一次独立的 HTTP API 调用，并不携带愚公的完整系统提示词上下文。如果工具端做了异步处理或开启了新的子进程，该进程不知道"要用中文思维"——这可能导致工具自身的日志/错误信息以英文输出。

验证方式：在工具执行层插入中文包装器，对所有输出加前缀/后缀，观察语言漂移率是否下降。

## 当前应对策略

1. **T0 已做**：6 处工具错误消息中文化（truncate.ts, tool.ts, invalid.ts, apply_patch.ts, bash.ts）
2. **T1 待做**：在工具执行层对非代码输出做中文包裹（需要在注册/processor 层加 hook）
3. **长期**：在 tool.execute.after 中检查输出语言倾向，若英文内容占比过高则补充中文提示重置语境

## 接受的上限

代码路径（`src/session/prompt.ts`）、类型错误（`TypeError: xxx is not a function`）、外部工具输出（npm/bun/git）本质上是英文的。强行汉化会丢失精确性并引入新的错误。**当前 ~90% 的中文保持率在大量英文工具输入的冲击下已是提示词工程的上限。**
