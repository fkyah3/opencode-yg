## ⚠️ TUI 崩溃调查（2026-04-28）

### 现象
- TUI 在非 TTY 模式下（bash 工具重定向输出）正常工作
- TUI 在真实终端（PowerShell）下立即崩溃退出，不报错误信息
- `--headless` 模式正常

### 诊断结果
通过逐步诊断标记确定崩溃点在 `createCliRenderer()` 内部：
1. `win32InstallCtrlCGuard()` ✅
2. `win32DisableProcessedInput()` ✅
3. `Terminal.getTerminalBackgroundColor()` ✅
4. `createCliRenderer()` — **原生崩溃** ❌（JS try/catch 无法捕获）
5. `@opentui/core` 的 `resolveRenderLib()` → `FFIRenderLib` 构造成功 ✅（DLL 加载正常）
6. `ziglib.createRenderer(80, 24, { testing, remote })` 返回合法指针 ✅
7. 崩溃发生在 `createRenderer` 返回后、`createCliRenderer` 返回前

### 验证
- 直接调用 Bun FFI `dlopen("opentui.dll")` → 成功
- 直接调用 `createRenderer(80, 24, 0)` → 返回非零指针
- `testing: true`（跳过 `setupTerminal()`）→ 仍然崩溃
- 非 TTY 下全链路正常 → TUI 的整体初始化逻辑没有结构性缺陷

### 上游线索
- [[Bug] opentui.dll panics during DllMain (#940)](https://github.com/anomalyco/opentui/issues/940)：DLL 静态初始化时 `invalid enum value` 恐慌，根因在 Bun 的 `RtlNtStatusToDosError` 类型声明（已向 Bun 提交修复）
- [[Windows Support #152]](https://github.com/anomalyco/opentui/issues/152)：Windows 综合跟踪（退出关闭终端、resize、鼠标、闪烁）
- 已知修复：`@opentui/core@0.1.100+` 含 Windows 代码签名，`v0.1.102` 更多修复

### 排查建议（给下一个 session）
1. **先试** `set OPENTUI_NO_GRAPHICS=1` 环境变量（opentui 原生支持）
2. **再试** `set TEMP=%LocalAppData%\Temp`（如果 TEMP 指向内存盘/非标准路径）
3. **最后试** 添加 Windows Defender 排除项（`%TEMP%` 目录）
4. 如果都不行，需要 opentui 团队介入调试 DLL crash dump
