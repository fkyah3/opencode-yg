为此仓库创建或更新 `AGENTS.md`。

目标是一份紧凑的指令文件，帮助未来的 OpenCode 会话避免错误并快速上手。每一行都应回答："没有帮助的话 agent 可能会漏掉这个吗？" 如果不会，就省略。

用户提供的关注点或约束（请遵守）：
$ARGUMENTS

## 如何调研

优先阅读价值最高的来源：
- `README*`、根清单、工作区配置、锁定文件
- 构建、测试、lint、格式化、类型检查、代码生成配置
- CI 工作流和 pre-commit / 任务运行器配置
- 已有的指令文件（`AGENTS.md`、`CLAUDE.md`、`.cursor/rules/`、`.cursorrules`、`.github/copilot-instructions.md`）
- 仓库本地的 OpenCode 配置，如 `opencode.json`

如果阅读配置和文档后架构仍不清楚，检查少量代表性代码文件以找到真实的入口点、包边界和执行流程。优先阅读解释系统如何连接在一起的文件，而非随机的叶子文件。

优先使用可执行的真相来源而非文字描述。如果文档与配置或脚本冲突，信任可执行来源，只保留你能验证的内容。

## 提取什么

寻找在该仓库中工作的 agent 最有价值的信息：
- 确切的开发命令，尤其是非显而易见的
- 如何运行单个测试、单个包或聚焦的验证步骤
- 有顺序要求的命令（如 `lint -> typecheck -> test`）
- monorepo 或多包边界、主要目录的归属，以及真实的应用程序/库入口点
- 框架或工具链的特殊之处：生成代码、迁移、代码生成、构建产物、特殊的环境加载、开发服务器、基础设施部署流程
- 与默认值不同的仓库特定风格或工作流约定
- 测试的特殊之处：fixture、集成测试先决条件、快照工作流、所需服务、不稳定或耗时的测试套件
- 有保留价值的既有指令文件中的重要约束

好的 `AGENTS.md` 内容通常是需要阅读多个文件才能推断出的、来之不易的上下文。

## 提问

仅当仓库无法回答某些重要问题时才向用户提问。最多使用 `question` 工具进行一轮简短提问。

好的问题：
- 未文档化的团队约定
- 分支/PR/发布期望
- 已知但未记录下来的缺失的设置或测试先决条件

不要询问仓库已明确说明的内容。

## 写作规则

只包含高价值、仓库特定的指导，例如：
- agent 可能猜错的确切命令和快捷方式
- 从文件名看不出的架构说明
- 与语言或框架默认值不同的约定
- 设置要求、环境特殊之处和操作陷阱
- 有参考价值的既有指令来源

排除：
- 通用的软件建议
- 冗长的教程或详尽的文件树
- 显而易见的语言约定
- 推测性声明或任何你无法验证的内容
- 更适合通过 `opencode.json` 的 `instructions` 引用存放在另一个文件中的内容

不确定时，省略。

优先使用简短的部分和项目符号。如果仓库简单，保持文件简单。如果仓库庞大，总结几个真正改变 agent 工作方式的结构性事实。

如果 `${path}` 处已存在 `AGENTS.md`，就地改进而非盲目重写。保留已验证的有效指导，删除冗余或过时的声明，并与当前代码库保持一致。
