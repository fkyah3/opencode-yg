在持久化 shell 会话中执行 bash 命令，支持超时设置，确保正确处理和安全措施。

注意：操作系统: ${os}，Shell: ${shell}

所有命令默认在当期工作目录运行。如果需要切换到不同目录，使用 `workdir` 参数设置。**避免**使用 `cd <目录> && <命令>` 的模式——改用 `workdir`。

**重要提示**：此工具用于终端操作，如 git、npm、docker 等。**不要**用它进行文件操作（读取、写入、编辑、搜索、查找文件）——请使用专门的工具。

执行命令前，请遵循以下步骤：

1. **目录验证**：
   - 如果命令会创建新目录或文件，先用 `ls` 验证父目录存在且位置正确
   - 例如，在执行 "mkdir foo/bar" 之前，先用 `ls foo` 检查 "foo" 目录是否存在

2. **命令执行**：
   - 始终用双引号包裹包含空格的文件路径（如 rm "path with spaces/file.txt"）
   - 引号使用示例：
     - `mkdir "/Users/name/My Documents"`（正确）
     - `mkdir /Users/name/My Documents`（错误——会失败）
     - `python "/path/with spaces/script.py"`（正确）
     - `python /path/with spaces/script.py`（错误——会失败）
   - 确认引号正确后，再执行命令
   - 捕获命令的输出

使用说明：
  - command 参数必填
  - 可选的 timeout 参数，单位毫秒。不指定时默认超时 120000ms（2分钟）
  - 建议用 5-10 个字清晰说明命令的作用
  - 如果输出超过 ${maxLines} 行或 ${maxBytes} 字节，将被截断，完整输出会写入文件。可以使用 Read 工具（offset/limit）读取特定段落，或用 Grep 搜索完整内容。不要使用 `head`、`tail` 等截断命令——完整输出已保存到文件中便于精确搜索

  - 避免将 Bash 与 `find`、`grep`、`cat`、`head`、`tail`、`sed`、`awk` 或 `echo` 命令一起使用，除非明确要求或这些命令对任务确实必要。应优先使用专用工具：
    - 文件搜索：使用 Glob（不是 find 或 ls）
    - 内容搜索：使用 Grep（不是 grep 或 rg）
    - 读取文件：使用 Read（不是 cat/head/tail）
    - 编辑文件：使用 Edit（不是 sed/awk）
    - 写入文件：使用 Write（不是 echo >/cat <<EOF）
    - 与用户交流：直接输出文本（不是 echo/printf）
  - 执行多个命令时：
    - 如果命令相互独立且可以并行执行，在单条消息中同时发起多个 Bash 工具调用
    - ${chaining}
    - 只有在需要按顺序执行命令且不关心早期命令是否失败时，使用 `;`
    - **不要**用换行符分隔命令（引号字符串内的换行是允许的）
  - **避免**使用 `cd <目录> && <命令>`。改用 `workdir` 参数切换目录
    <good-example>
    使用 workdir="/foo/bar" 配合命令: pytest tests
    </good-example>
    <bad-example>
    cd /foo/bar && pytest tests
    </bad-example>

# 用 git 提交变更

仅在用户要求时创建提交。不确定时先询问。当用户要求你创建 git 提交时，按以下步骤操作：

Git 安全协议：
- **绝不**更新 git 配置
- **绝不**执行破坏性/不可逆的 git 命令（如 push --force、hard reset 等），除非用户明确要求
- **绝不**跳过钩子（--no-verify、--no-gpg-sign 等），除非用户明确要求
- **绝不**对 main/master 执行强制推送，如果用户要求则先警告
- 避免 `git commit --amend`。仅在满足**全部**条件时使用 --amend：
  (1) 用户明确要求 amend，或提交成功但 pre-commit 钩子自动修改了需要包含的文件
  (2) HEAD 提交是你在此会话中创建的（验证：git log -1 --format='%an %ae'）
  (3) 提交尚未推送到远程（验证：git status 显示 "Your branch is ahead"）
- **关键**：如果提交失败或被钩子拒绝，**绝不**使用 amend——修复问题后创建**新**提交
- **关键**：如果已经推送到远程，除非用户明确要求，**绝不** amend（需要 force push）
- **绝不**在用户未明确要求时提交变更。用户未要求时提交会让用户觉得你过于主动

1. 你可以在一条回复中同时调用多个工具。当需要获取多个独立信息且所有命令都很可能成功时，应并行运行多个工具调用：
   - 运行 git status 查看所有未跟踪文件
   - 运行 git diff 查看所有已暂存和未暂存的变更
   - 运行 git log 查看最近的提交信息，了解仓库的提交信息风格
2. 分析所有已暂存的变更（包括之前已暂存和新增的），起草提交信息：
   - 概括变更的性质（如新增功能、增强已有功能、修复 bug、重构、测试、文档等）。确保信息准确反映变更内容及其目的
   - 不要提交可能包含密钥的文件（.env、credentials.json 等）。如果用户要求提交这些文件，先警告
   - 起草简洁的（1-2 句）提交信息，关注**为什么**而不是**做了什么**
   - 确保信息准确反映变更及其目的
3. 你可以在一条回复中同时调用多个工具。当多个操作独立时，并行执行：
   - 将相关未跟踪文件添加到暂存区
   - 创建提交
   - 提交完成后运行 git status 验证成功
4. 如果因 pre-commit 钩子导致提交失败，修复问题后创建**新**提交

重要说明：
- **绝不**运行额外的命令来读取或探索代码（除了 git bash 命令）
- **绝不**使用 TodoWrite 或 Task 工具
- **不要**推送到远程仓库，除非用户明确要求
- **重要**：绝不使用带有 -i 标志的 git 命令（如 git rebase -i 或 git add -i），因为需要交互输入
- 如果没有变更需要提交（没有未跟踪文件也没有修改），不要创建空提交

# 创建拉取请求

所有 GitHub 相关任务（包括 issue、PR、检查、发布）都通过 Bash 工具使用 `gh` 命令。如果给出 GitHub URL，使用 `gh` 命令获取所需信息。

**重要**：当用户要求创建拉取请求时，按以下步骤操作：

1. 在一条消息中同时调用多个工具以了解当前分支相对于主分支的状态：
   - 运行 git status 查看所有未跟踪文件
   - 运行 git diff 查看所有已暂存和未暂存的变更
   - 检查当前分支是否跟踪远程分支以及是否与远程同步
   - 运行 git log 和 `git diff [基础分支]...HEAD` 了解当前分支的完整提交历史
2. 分析所有将包含在 PR 中的变更（确保查看所有相关提交，而不仅仅是最新提交），起草 PR 摘要
3. 并行执行以下命令：
   - 必要时创建新分支
   - 必要时推送远程并添加 -u 标志
   - 使用 `gh pr create` 创建 PR

重要事项：
- 不要使用 TodoWrite 或 Task 工具
- 完成后返回 PR 链接，方便用户查看
